Step-by-Step Walkthrough
1
Browser → middleware.ts GET /dashboard
Session guard runs before anything else
Runs at the Edge (or Node.js) runtime — before any React rendering occurs.

Creates a createServerClient wired to request cookies. Calls supabase.auth.getClaims() — this validates the JWT and silently refreshes it if expired, writing updated cookies into the response.

Decision:
• Path is in PUBLIC_PATHS (/, /login, /signup) or starts with /api/ → allow through.
• No session + protected path → redirect(/login).
• Has session + visiting /login or /signup → redirect(/dashboard).
• Has session + protected path → NextResponse.next() with refreshed cookies.
Cookies are forwarded to every subsequent layer via the response headers
2
Server Component app/layout.tsx renders on the server
Root layout — pure server, no hooks
layout.tsx has no 'use client' directive — it is a React Server Component. It runs on the server for every navigation to a new route.

Its only job is to provide the HTML shell and mount the two client boundary providers:

<AppProviders> — establishes the React Query + Auth context tree.
<AppLayoutShell> — decides sidebar vs. full-page.
<Toaster /> — sonner toast container.

Why Server Component? It can stream the initial HTML to the browser immediately, and it can pass Server Component children (page.tsx) down as pre-rendered RSC payloads — even though the layout shell around them is client-side.
Server renders the shell HTML; client components hydrate on the browser
3
Client Component AppProviders.tsx hydrates in browser
Provider tree — QueryClient + AuthProvider
AppProviders is the client boundary — everything inside it runs in the browser.

QueryClientProvider — creates a TanStack React Query QueryClient (one per page load, stable via useState). Makes useQuery / useMutation available to all descendants.

AuthProvider (AuthContext) — on mount:
1. Calls supabase.auth.getSession() → reads the session cookie already in the browser (placed there by middleware or the callback route). Sets user, session state.
2. Subscribes to supabase.auth.onAuthStateChange() for future events (token refresh, sign out).
3. Once user is known, queries tenant_members for { tenant_id, role } and stores as tenantMember.
Sets loading = false when done.
useAuth() is available to any component inside AppProviders
4
Client Component AppLayoutShell.tsx — sidebar or passthrough
Route-aware layout switcher
Uses usePathname() to read the current URL on the client.

Imports PUBLIC_PATHS directly from middleware.ts — single source of truth for which routes are public. No duplication.

If path is in PUBLIC_PATHS (/login, /signup, /):
  → Returns {children} only — no sidebar, full-screen page.

Otherwise (authenticated pages):
  → Returns flex h-screen container with:
    • <AppSidebar /> — fixed left column (w-64)
    • <main> — scrollable right column with page content
PUBLIC_PATHS imported from middleware.ts — one list rules both server guard and client layout
5
Client Component AppSidebar.tsx renders
Sidebar — role-filtered nav + org brand
Reads auth state from useAuth(): tenantMember, tenantId, user, signOut.

Nav filtering: navItems is a static array of routes each with an allowed roles[] list. Filtered by tenantMember.role (defaults to 'partner' if undefined). Only routes the user's role can access are rendered.

Active state: usePathname() compared against each nav item's path. Matching item gets the accent background style.

OrgBrand sub-component: Two chained React Query fetches:
  1. ['tenant', tenantId] → supabase.from('tenants').select('id, name, name_abbreviation, logo_url')
  2. ['org-logo-signed-url', logoPath] → getLogoSignedUrl() (only if logo_url exists)
Displays logo image or derived initials (e.g. "NP" from "NSP Pay") as fallback.

Sign out: Calls signOut() from AuthContext, then router.push('/login').
Nav items are filtered per role — a 'partner' sees fewer items than 'owner'
6
Page page.tsx renders as children inside <main>
The actual page content
Rendered as children passed down through the layout tree. Can be a Server Component (default in App Router — no 'use client') or a Client Component.

Server Component pages can do direct Supabase calls via lib/supabase/server.ts, use await cookies(), or call Server Actions.

Client Component pages can use useAuth(), useQuery(), and browser APIs.

Rendered inside <main className="flex-1 overflow-y-auto"> — scrollable, full height, with a container max-w-7xl py-8 wrapper.
React Component Tree (authenticated page)
<RootLayout> Server Component — layout.tsx
<AppProviders> 'use client' — QueryClientProvider + AuthProvider
<QueryClientProvider> TanStack React Query
<AuthProvider> user, session, tenantMember, tenantId, signOut
<AppLayoutShell> 'use client' — reads usePathname()
<AppSidebar> 'use client' — useAuth(), usePathname(), useQuery()
<OrgBrand tenantId> useQuery → tenants table → signed logo URL
<main> flex-1 scrollable area
{children} page.tsx — Server or Client Component
<Toaster /> sonner toast portal — outside layout shell
AppSidebar Render Sequence

On initial mount

1. Hooks read from AuthContext
useAuth() → tenantMember, tenantId, user, signOut. AuthContext may still be loading=true on first paint — sidebar renders with userRole = 'partner' fallback until resolved.
2. Nav items filtered by role
Static navItems[] filtered: item.roles.includes(userRole). Result is the subset of links this user may see. Roles: owner > admin > manager > partner.
3. OrgBrand fires React Query
Query ['tenant', tenantId] — fetches name, name_abbreviation, logo_url from Supabase. Only enabled once tenantId is non-null.
4. Logo signed URL query (chained)
Query ['org-logo-signed-url', logoPath] — runs only if logo_url exists. Calls getLogoSignedUrl() to get a short-lived presigned URL from Supabase Storage.
5. Renders final state
Shows logo image (or initials fallback), org name with Tooltip, filtered nav buttons, user email footer, and Sign out button.

On route change (client navigation)

Middleware does NOT re-run
Client-side navigation (e.g. router.push('/providers')) does not trigger the middleware — middleware only runs on full HTTP requests. The sidebar itself handles route changes.
usePathname() re-evaluates
usePathname() in both AppLayoutShell and AppSidebar automatically updates. AppLayoutShell re-checks PUBLIC_PATHS. AppSidebar re-computes isActive for each nav button.
React Query cache reused
Tenant name + logo queries are already cached. OrgBrand re-renders instantly with cached data — no new network requests on every navigation.
New page.tsx streams in
Next.js fetches only the new page's RSC payload — the layout shell (sidebar, providers) stays mounted. No full re-render of the sidebar.
Key Design Decisions

Why AppLayoutShell is a Client Component

  • ▸Needs usePathname() — a React hook — to decide whether to show the sidebar. Hooks require 'use client'.
  • ▸Imports PUBLIC_PATHS from middleware.ts — single source of truth so the server guard and client layout always agree on what's public.
  • ▸The Server Component root layout (layout.tsx) passes children through, so individual pages can still be Server Components even though the shell is client-side.

Why there is no server-side sidebar data fetch

  • ▸Tenant name and logo are fetched client-side via React Query. This keeps the RSC streaming fast — the shell HTML arrives before any DB call completes.
  • ▸React Query caches results across navigations — the sidebar doesn't re-fetch on every route change, only on cache expiry.
  • ▸The signed logo URL is a separate chained query so it can be independently refreshed when it expires, without refetching the tenant metadata.
